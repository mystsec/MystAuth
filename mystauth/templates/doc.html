Intro

The Myst Auth API allows you to add secure, customizable passkey authentication and identity management to your web app with just a few lines of code! Passkeys let users authenticate using their device's biometrics or a security key, eliminating the need to remember passwords. Using asymmetric cryptography, passkeys also greatly reduce the threat of phishing attacks, achieving better security than passwords with a better user experience. Although passkeys are relatively new, 96% of users' devices/browsers support a version of passkeys used by Myst Auth. Learn more about passkeys in this Google Blog and this Yubico explainer.

You can try a demo of Myst Auth's user sign up/login experience by signing in to get your Myst Auth API Key at the start of the Quickstart section. By default, your account gets 100 API Tokens (100 user accounts), but if you are using Myst Auth for a non-profit or educational resource, please contact us (contact@mystauth.com) and we may be able to upgrade your account for free.

Quickstart #

You can add Myst Auth to your site in 3 steps:

  1. Get Myst API Key
  2. Add Myst Auth Links to Frontend
  3. Add Authentication Middleware to Backend

Get Myst API Key:

Sign Up / Log In to try out Myst Auth and get your API Key! You will be able to manage your API in the dashboard once logged in.

Once Signed In, you'll get a Request ID (rid). Your Request ID is used to provide a single base Auth Link (which you'll use next) to initiate user sign ups or logins.


Add Myst Auth Links to Fronted:

Your Auth Links are where your user is directed so they can securely sign up or login to your app. It follows this format:

https://mystauth.com/auth/?rid=your-request-id&ref=your-redirect-link

The ref url parameter must be set and should point to the page (on your domain) which you want the user to reach after they sign up or login. For sign ups this would typically be a user onboarding page where you can add the user to your database. To protect Auth Tokens, your redirect link must use TLS and HTTPS and it must use the origin linked to your API Key (Myst Auth blocks requests which do not meet these requirements).

If your redirect link has its own parameters, you will need to encode the url before using it in a Auth Link. Learn more about this and other parameters which can be use for UI customization in the Auth Link section.

Now you can add (either hard coded or dynamically generated) your Auth Links to your frontend so that your users can authenticate themselves. Here is an example Sign Up button in HTML:

<button onclick="window.location.href='https://mystauth.com/auth/?rid=your-rid&ref=redirect-link'">Sign Up</button>

Once your user authenticates with Myst, they will be redirected to your page with their Username and a temporary Auth Token (both as url parameters) which you will use to verify their identity by making an API call to Myst Auth's token verify service. If the token is valid, the API will return a new token which you must pass back to the user so they can remain logged in. This authentication process can be added to all of your pages and APIs through an authentication middleware.


Add Authentication Middleware to Backend:

The token verification API has the following endpoint:

https://mystauth.com/api/v1/user/token/verify/

with the following POST parameters:

Parameter Description
id Your API id.
apiKey Your API key.
usr User's username
token User's Auth Token

If the token is verified successfully, the JSON response will follow this format:

{success: true, token: the-new-auth-token}

The Auth Tokens generated by Auth Links timeout after 45 seconds to prevent Man-In-The-Middle attacks, but new tokens generated after the initial authentication process timeout after 1 hour by default (can be changed in your API account settings). Learn more in the Auth Token section.

Here is an example Authentication Middleware written in python for the Django framework:

#Django Myst Auth Middleware Example
import requests

class MystAuthMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response  #initiate middleware, not special for Myst Auth

    def __call__(self, request, *args, **kwargs):
        if request.method == 'GET' and 'usr' in request.GET and 'token' in request.GET:  #check if relevant url params exist
            user = request.GET.get('usr')  #get username param
            token = request.GET.get('token')  #get auth token param

            myst_endpoint = "https://mystauth.com/api/v1/user/token/verify/" #Myst Auth Token verify endpoint

            myst_params = {  #POST parameters
                "id": myst_id,  #your API id, best to retrieve from environment variable or secrets file
                "apiKey": myst_api_key,  #your API key, best to retrieve from environment variable or secrets file
                "usr": user,
                "token": token,
            }

            auth = requests.post(myst_endpoint, json=myst_params)  #make API call, pass params
            auth = auth.json()  #get json

            authenticated = auth['success']  #gets 'success' field from json

            if authenticated:  #checks if 'success': true, token was verified
                request.authenticated = True  #sets request param 'authenticated' to True
                request.token = auth['token']  #sets request param 'token' to the new token from the API call json
            else:
                request.authenticated = False #sets request param 'authenticated' to False
        else:
            request.authenticated = False #sets request param 'authenticated' to False

        return self.get_response(request)  #returns authenticated request to views.py in Django (the page's controller function)

There are many ways for your middleware to let your controller function know the result of the authentication, but in this example, the following request parameters are set: authenticated informing the result, and, if it was successful (the user's Auth Token is valid and has not timed out), token providing the new Auth Token. Once the user has been authenticated through the Auth Link, you can choose to continue using Myst Auth Tokens to manage their session or you can use your own session management system. To continue using Myst to secure your user, the new token must be returned to the user (ex. through a hidden template element or JSON response field) so that they can be authenticated in their next request or call to your API.

Once your user gets their Auth Token, you can choose to store the token in a cookie or localStorage so all the user's tabs to your page can remain authenticated through the middleware. If you use cookies, you will need to modify your middleware to be able to get the username and Auth Token from the cookie in addition to url params, and if you use localStorage, you will need to add more code to your frontend to check for the token and refresh the page to the authenticated session. Each request made by users to your app will authenticate their token with Myst Auth and the new token is returned.

API Settings #

When you create a new API account to use Myst Auth and in your API account dashboard, you can change the following attributes:

Attribute Description
Origin The Hostname of your app (includes subdomain and domain). Learn more about Hostnames. Must match the hostname of the redirect link you set in Myst Auth Links. Your account origin is editable.
TTL The time-to-live for Auth Tokens generated after the initial passkey authentication. Default is 3600 seconds (1 hour). TTL for initial Auth Tokens passed to your redirect link after the Myst Auth Link passkey authentication is 45 seconds and is not editable.
BioOnly If set to True, enforces that users must register their passkey using on-device biometric authentication (Apple TouchID, Windows Hello, etc.) and are not allowed to use external authenticators like security keys. Default is False. BioOnly is editable.
AllowResets If set to True, allows you to offer passkey resets/account recovery to users. Default is False for security purposes. AllowResets is editable. Learn more in the Reset Link section.

Auth Link #

Myst Auth Links are where you direct your users when they are signing up or logging in to your app so they can authenticate through Myst Auth. The base url is:

https://mystauth.com/auth/

with the following url/GET parameters:

Parameter Description
rid Your Request ID, listed in your API account dashboard.
ref The redirect url where the user should be directed after being authenticated. Must use HTTPS and match the origin registered with your API account. If your redirect url itself contains url parameters, it must be URL Encoded (in JS using encodeURIComponent()) for those params to be properly passed. Otherwise, the url does not need to be encoded, but should be the last parameter.
usr (Optional) Username to be pre-filled in the login form. Meant to be used for cases such as Auth Token timeouts where the user is known and must be directed to the Myst Auth login page to re-authenticate. Note: This parameter should only be used in conjunction with the #login parameter appended to the url (learn more below).
rst (Optional) The url where the user should be directed to recover their account if their passkey is lost. Must use HTTPS and match the origin registered with your API account. If this link itself contains url parameters, it must be URL Encoded (in JS using encodeURIComponent()) for those params to be properly passed. Learn more about passkey resets in the Reset Link section.
img (Optional) ID of image to be set as background image. See below for list of available IDs and images.
bgclr (Optional) Hex color, without '#', for the UI background. If a background image is set, this param won't be used.
clr (Optional) Hex color, without '#', for the Myst Auth UI's button.
hovclr (Optional) Hex color, without '#', for the Myst Auth UI's button hover.

The img, bgclr, clr, and hovclr parameters were added to Myst Auth to help you convey your website's theme on the Myst Auth page UI and give your user's a more familiar experience. Contact us if you want to add custom branding to your Auth Link.

Additionally, if you are using an Auth Link for a login button (you know the user has an account), you can append the #login url hash parameter so the UI will show the login option first. And, if you know the user's username (ex. you are redirecting them to login after a session timeout), you can use the usr param so that the user doesn't need to enter it themselves.

Putting these options together, here is an example Auth Link with a background image and setting the login tab to be shown first:

https://mystauth.com/auth/?rid=00000000000000000000000000000000&img=RdELgb1bNz8&ref=https%3A%2F%2Fexample.com%2Fdashboard%2F%3Fa%3D1%26b%3D2#login

Auth Token #

Myst Auth Tokens verify a user's identity after they have been authenticated through their passkey. To verify a user's Auth Token, use the following endpoint:

https://mystauth.com/api/v1/user/token/verify/

with the following POST parameters:

Parameter Description
id Your API id.
apiKey Your API key.
usr User's username
token User's Auth Token

The API will return one of the following responses:

JSON Response Description
{success: true, token: the-new-auth-token} The token was valid and has not timed out. A new token is returned.
{'success': false, 'info': 'Login Timed Out!'} Authentication failed, the token (session) timed out.
{'success': false, 'info': 'Authentication Failed!'} Authentication failed, the token is not valid or has already been used.
{'success': false, 'info': 'API Authentication Failed!'} Authentication failed, your API id and key did not authenticate successfully.

Once an Auth Token is used to authenticate, it is deleted and the new token is returned. If you choose to implement/continue using your own session management system to handle indentity verification after the user has authenticated through Myst, you do not need to do anything with the new token once the original is verified. Otherwise, you will need to pass the new token to the user/client (ex. through a hidden template element or JSON response field). The client script can store the Auth Token in a cookie or localStorage so that all the user's tabs to your app can be authenticated through your middleware which calls the Myst Auth API to verify the token. In addition to authenticating pages, you can use Auth Tokens and the same middleware to secure your API endpoints.

If the Auth Token verification success returns false, you should log the user out in the frontend (direct to login Auth Link if their token timed out and direct them, for ex., to your home page if their authentication failed), and ensure you don't send any user data to the un-authenticated session.

User Management #

When a user signs up through Myst Auth, they must use a unique (within users of your app) username in order to secure their passkey and to prevent Myst from requiring a different unique (and personal) identifier like an email address or phone number. At registration, Myst Auth enforces that the username contains only alphanumeric characters, underscores, and hyphens (no spaces allowed).

After the user signs up and is redirected to your onboarding page, you should register their unique username in your database and associate their data with that unique username.

At this time, we have not added a method for changing a user's username associated with their passkey, however you can implement a display name system for your users to change their public/visible username.

If a user wants to delete their account, the following Myst Auth endpoint can be used to delete all data associated with them (this is permanent and they will be unable to use their passkey in the future):

https://mystauth.com/api/v1/user/delete/

with the following POST parameters:

Parameter Description
id Your API id.
apiKey Your API key.
usr User's username
token User's Auth Token

The API will return one of the following responses:

JSON Response Description
{success: true} The token was valid and has not timed out. The account was successfully deleted.
{'success': false, 'info': 'Login Timed Out!'} Authentication failed, the token (session) timed out. The account was not deleted.
{'success': false, 'info': 'Authentication Failed!'} Authentication failed, the token is not valid or has already been used. The account was not deleted.
{'success': false, 'info': 'API Authentication Failed!'} Authentication failed, your API id and key did not authenticate successfully. The account was not deleted.

Reset Link #

Most users' passkeys are automatically backed up through their device's account (iCloud Keychain, Google Account, Outlook Account, etc.), however there is still a chance that the user loses their device and/or access to their passkey for your app. So, Myst Auth provides an optional passkey reset system which can be enabled by changing the allowReset setting in your Myst Auth Account Dashboard to True.

NOTE: ALLOWING USERS TO RESET THEIR PASSKEY SIGNIFICANTLY INCREASES THE RISK AND POSSIBILITY OF ACCOUNT COMPROMISING ATTACKS (LIKE PHISHING)! THIS SETTING SHOULD NOT BE ENABLED ON APPLICATIONS WHERE SECURITY IS CRITICAL FOR PROTECTING SENSITIVE DATA!

In order to offer passkey resets securely to a user, you must have a VERIFIED communication method (such as an email address which the user has verified) attached to their Myst Auth username (their communication method should be collected and verified during the user onboarding process after they first sign up through the Myst Auth login portal). Once you have a system for verifying a user's communication method, you can offer a page where your users enter their communication details (ex. enter their email address), and if the it matches (is linked in your database to) a Myst Auth username/account, you can send to the user (via that communication method) a Myst Auth Reset Link. Your 'request reset' page can also be included in the Myst Auth Link parameters, allowing a Lost Passkey? prompt to be shown to your users directly on the Myst Auth login portal.

The allowReset = True setting allows you to call the following Myst Auth endpoint in order to generate a Magic Code which is used to provide the Reset Link:

https://mystauth.com/api/v1/user/reset/

with the following POST parameters:

Parameter Description
id Your API id.
apiKey Your API key.
usr User's username

The API will return one of the following responses:

JSON Response Description
{'success': True, 'mcode': the-magic-code-used-in-the-reset-link} A Magic Code was successfully generated and expires after 10 minutes.
{'success': false, 'info': 'API Authentication Failed!'} Authentication failed, your API id and key did not authenticate successfully. The Magic Code was not set.
{'success': False, 'info': 'allowReset set to False, change in API Account Dashboard'} Your account permissions have not been set to allow the creation of Reset Links (see above).

Once a user requests a Reset Link by entering a valid and verified communication method to your 'request reset' page, you must call the /user/reset/ API endpoint above to establish a Magic Code, so you can send the following Reset Link to the user's verified communication method:

https://mystauth.com/reset/

with the following url/GET parameters:

Parameter Description
rid Your Request ID, listed in your API account dashboard.
ref The redirect url where the user should be directed after their passkey is reset. Must use HTTPS and match the origin registered with your API account. If your redirect url itself contains url parameters, it must be URL Encoded (in JS using encodeURIComponent()) for those params to be properly passed. Otherwise, the url does not need to be encoded, but should be the last parameter.
mc A valid Magic Code you generated on reset request from the user.
usr The Username of the user who is trying to reset their passkey.
img (Optional) ID of image to be set as background image. See the Auth Link section for more details.
bgclr (Optional) Hex color, without '#', for the UI background. If a background image is set, this param won't be used.
clr (Optional) Hex color, without '#', for the Myst Auth UI's button.
hovclr (Optional) Hex color, without '#', for the Myst Auth UI's button hover.

The Reset Link uses mostly the same parameters as the Auth Link, but adds the Magic Code to verify the reset session. A Magic Code is valid for the user to open the Reset Link for 10 minutes after it is generated. Once the Reset Link is opened, the Magic Code becomes invalid (so the same link cannot be used again) and the user has another 10 minutes to begin the new passkey registration process. Once the registration process is started, the user has 3 minutes to complete the process (ex. put their finger on their device's fingerprint scanner, etc.) or cancel their device's registration prompt. If the device passkey process is cancelled, the user will have another 3 minutes to restart it. After the reset process is complete, the user is redirected to the url you set along with an Auth Token, which has a 45 seconds TTL like every other Myst Auth Token sent after authentication.

These measures were implemented to make the reset process as secure as possible, but it is important to note that anyone with a Reset Link with a valid Magic Code has the ability to reset the account and takeover the account. This means that you are placing trust in the user's communication method and any compromise of it will be a compromise of the authentication system when Reset Links are enabled. We are working on adding email support and end-to-end handling of the reset process natively in Myst Auth.

Security #

Account Settings

Your account settings can be changed from your Myst Auth Account Dashboard, and these processes are protected by both your API key and your passkey session authentication.

We STRONGLY recommend you cycle your Myst Auth API key from the account dashboard regularly (once per month or during scheduled maintainence windows) to prevent API compromise and ensure your account uses our latest security updates. While a compromised API key is a serious security breach, the attacker will not be able to change critical settings (such as allowReset) or cycle or delete your API key thanks to the passkey verification required to access your acocunt dashboard. If you have not enabled Reset Links, the attacker will only be able to impact user accounts if they are also able to intercept a valid user Auth Token.

Hashing

Myst Auth currently uses PBKDF2-HMAC-SHA512 with 1,000,000 iterations (as per recommendations by OWASP and NIST) to hash API keys, Auth Tokens, and Magic Codes for storage in our database (learn more about password hashing from OWASP). We have added internal support for Argon2id hashing and are working on validating the security of our setup before allowing it as an alternative hash option.
